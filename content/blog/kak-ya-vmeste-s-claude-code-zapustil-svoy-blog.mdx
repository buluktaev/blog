Продуктовый дизайнер может запустить полноценный блог с автоматизированным пайплайном публикации, не написав ни строчки кода вручную. Для этого достаточно Claude Code Skills, шаблона Astro Paper и промптов — вместо ручной работы со стеком. Этот пост — и есть доказательство: он создан, отформатирован и опубликован через тот самый пайплайн, о котором в нём рассказывается.

## Я не пишу код — я пишу промпты

Меня зовут Булат, я продуктовый дизайнер. Работаю в рекрутинговом агентстве над внутренней ATS-системой для найма IT-специалистов. До этого строил фичи в стартапах: Quiz Lab — образовательные квизы, и Sowaa — умный матрас с анализом фаз сна и мобильным приложением.

Я не программист и не называю себя разработчиком. Мой инструмент — промпт, не IDE.

Это мой первый пост в блоге. И он — о том, как сам блог появился. Мета? Возможно. Но по опыту, лучший способ объяснить процесс — показать его на себе.

## Проблема: между мыслью и публикацией — пропасть

У меня давно копились заметки. За несколько месяцев в Notion накопилось больше сорока заметок о работе с агентами. Наблюдения о дизайн-процессе, о том, как продуктовое мышление меняется, когда рядом есть Claude. Мыслей хватало. Не хватало инфраструктуры — конкретно: нигде не было места, куда эти мысли могли бы попасть без боли.

Между «есть черновик в заметках» и «пост опубликован» лежит неприятная долина:

- Выбор стека и шаблона
- Настройка деплоя
- Форматирование Markdown с правильным frontmatter
- Git-коммиты, пуши, CI/CD
- И главное — вопрос голоса: как не потерять себя, когда AI помогает с текстом

По данным McKinsey за 2025 год, большинство маркетологов уже используют AI для ускорения контента, но главным барьером называют потерю голоса. Скорость не проблема. Аутентичность — проблема.

Мне нужно было решить обе задачи одновременно: убрать техническое трение и сохранить голос.

## Решение: Astro Paper + Claude Code Skills + один промпт

### Почему Astro Paper

Astro Paper — это готовый шаблон блога: Markdown + TypeScript + Tailwind. Lighthouse стабильно показывает ~100. Деплой на Vercel — без ручной CI-конфигурации. Попросил агента склонировать репозиторий, настроить конфиг — и статика готова.

Ключевое: я не «поднимал» блог. Я описал, каким он должен быть, а агент сделал остальное.

### Claude Code Skills — автоматизация без магии

Claude Code Skills — механизм, доступный с декабря 2025. Идея элегантная: файл SKILL.md с progressive disclosure. При старте агент видит ~100 токенов — короткое описание. При активации разворачивается полная инструкция — ~2000 токенов с шаблонами, правилами, примерами.

Я собрал скилл для контент-пайплайна. Вот как он работает:

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  Черновик    │────▶│  Claude Code  │────▶│  Markdown   │
│  (заметки)  │     │  Skill        │     │  + front-   │
│             │     │  /blog-post   │     │  matter     │
└─────────────┘     └──────┬───────┘     └──────┬──────┘
                           │                     │
                    ┌──────▼───────┐     ┌──────▼──────┐
                    │  Проверка    │     │  git commit  │
                    │  голоса и    │     │  + push      │
                    │  структуры   │     │  → Vercel    │
                    └──────────────┘     └─────────────┘
```

Одна команда — и агент берёт черновик, структурирует его по плану, проверяет тон, генерирует frontmatter, коммитит и пушит. Vercel подхватывает — пост в сети.

### Три принципа, на которых стоит пайплайн

Первое: агент не автор — агент инструмент. AI не придумывает за меня, о чём писать. Я прихожу с заметками, тезисами, конкретными фактами. Агент структурирует, форматирует, проверяет. Разница — как между «нарисовать картину» и «натянуть холст на подрамник». Оба шага нужны, но творческий акт — в первом.

Второй момент — голос фиксируется в конфиге, а не в надежде. В скилле прописан writing guide — синтаксические паттерны, антипаттерны, replacement patterns[^1]. Если агент пишет «важно понимать» — скилл это перехватывает. Если появляется менторский тон — конфиг его отсекает. Голос — не случайность, а система правил.

И наконец: progressive disclosure экономит контекст. Скилл не загружает 2000 токенов инструкций при каждом запуске Claude Code. При старте — 100 токенов, краткое описание. Полная инструкция разворачивается только при вызове `/blog-post`. Это не мелочь — в длинных сессиях контекстное окно на вес золота.

## Как это выглядит на практике

Я открываю Claude Code. У меня есть черновик — сырые мысли, факты, пара цитат. Я даю промпт:

> Возьми черновик ниже. Не переписывай — только структурируй по плану:
> [вставить структуру поста].
> Сохрани мой тон: короткие предложения, без корпоративного языка,
> от первого лица. Если чего-то не хватает — спроси, не выдумывай.
>
> ЧЕРНОВИК:
> [...]

Обратите внимание на последнюю строку: «спроси, не выдумывай». Это критически важно. AI-модели склонны к галлюцинациям — особенно когда чувствуют пробел, который можно «заполнить». Явный запрет на выдумку — часть дизайна промпта, а не вежливость.

Агент возвращает структурированный Markdown. Я читаю, правлю — обычно это 10-15 минут. Потом агент коммитит, пушит. Vercel деплоит. Готово.

Весь цикл — один сеанс. Никакого переключения между инструментами, никакого «сначала напишу, потом разберусь с деплоем».

## О голосе и perplexity

Есть две метрики, которые используют для оценки «человечности» текста: perplexity и burstiness. Perplexity — насколько предсказуем следующий токен. Burstiness — разброс длины предложений.

AI-тексты обычно ровные — средняя длина предложения стабильная, слова предсказуемые. Человеческие тексты рваные: короткое предложение. Потом длинное, с подчинёнными конструкциями, уточнениями в скобках, внезапной сменой ритма — и снова короткое.

Writing guide в моём скилле намеренно задаёт burstiness: «короткие утверждения чередуются со сложными конструкциями», «em-dash для уточнений», «юмор спрятан в скобках». Это не стилистический каприз — это инженерное решение для аутентичности[^2].

По опыту — голос не возникает из ниоткуда. Его нужно формализовать. Парадокс: чтобы текст звучал естественно, правила должны быть записаны явно. Иначе каждый новый промпт — лотерея.

## Результат

Этот пост — результат пайплайна. Lighthouse показал 100. На сборку от черновика до публикации ушёл один сеанс — не день, не неделя. Голос остался моим.

## О голосе и автоматизации: почему это не одно и то же

Самый частый страх, который я слышу от коллег: «AI будет писать за меня, и это будет не моё». Справедливо — если дать AI пустой промпт и попросить «напиши пост про X». Тогда получишь усреднённый текст, написанный для всех и ни для кого.

Но если ты приходишь с мыслями, фактами, голосом — AI становится мультипликатором, а не заменой. Убирает трение между идеей и публикацией. Берёт на себя стек, форматирование, деплой — всё то, что не имеет отношения к тексту, но без чего текст не увидит читатель.

Я дизайнер. Я проектирую опыт — в том числе свой собственный опыт создания контента. Claude Code Skills — это способ задизайнить пайплайн так же, как я дизайню интерфейс: с системой, с правилами, с уважением к пользователю. Только пользователь здесь — я сам.

Следующий пост — о том, как устроен сам скилл изнутри. Файловая структура, логика progressive disclosure, конкретные промпты. Если интересно — вы знаете, где найти.

[^1]: Replacement patterns — это пары «если агент написал X → замени на Y». Например: «написал код» → «попросил агента написать». Простой механизм, но он радикально меняет ощущение от текста.

[^2]: Забавно: чтобы текст не казался написанным AI, нужно дать AI больше инструкций, а не меньше. Контринтуитивно, но работает.
